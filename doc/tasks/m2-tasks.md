# Snow M2 — 分批任务清单（草稿）

> 日期：2026-04-10  
> 版本：v0.1  
> 状态：讨论中  
> 原则：**先把 Snow 从 CLI 核心引擎推进到可部署的 Web 产品，再扩展消息能力与调度能力**

---

## M2 目标

M2 的重点不是继续堆更多人格能力，而是把 M1 已经跑通的核心循环真正产品化：

1. 提供可部署的 Web 端（Next.js + Vercel）
2. 把进程内 `setTimeout` 调度替换为真正可用的延时队列
3. 支持主动消息
4. 支持图片消息
5. 在推进这些能力时，顺带消化对应的 M1 技术债

---

## M2 不做的事

M2 先不展开以下方向：

- 多平台接入（QQ / Telegram / 微信）
- MCP / Skill / Capability 系统
- 多角色系统
- 图片生成
- 复杂社交或社区功能

M2 只聚焦：**Web 化、Serverless 化、主动消息、图片消息、核心链路稳定化**

---

## Batch 1：Web 端 + Vercel 部署 + 延时队列改造

### 目标

创建 `packages/web`，接入 `@snow/core`，并完成一个真正可用的 Web 版本：

- 可以部署到 Vercel
- 可以登录 / 注册
- 可以和 Snow 聊天
- 可以在 Web / Serverless 环境中正常触发完整异步后处理

### 任务

1. 创建 `packages/web`（Next.js App Router）
2. 接入 `@snow/core`
3. 实现最小登录 / 注册功能
4. 实现 Web 聊天页面
5. 实现 `/api/chat`
6. 支持流式响应
7. 配置 Web 端环境变量
8. 验证 Vercel 可部署
9. 用 Upstash QStash 替换 `packages/core/src/scheduler/delayed-task.ts`
10. 提供 Idle 回调入口（HTTP / Route Handler）
11. 重构“新任务覆盖旧任务”的逻辑
12. 保证同一用户始终只有一个待执行的会话结束任务
13. 保证 Idle 任务幂等
14. 保证失败可重试
15. 让 Web / Vercel 环境可正常触发：
   - 记忆提取
   - 关系更新
   - 摘要持久化
   - 情绪快照
   - 趋势摘要

### 职责边界

Batch 1 中必须保持以下边界：

- `scheduleDelayedTask()` / `cancelDelayedTask()` / `executeIdleTasks()` 仍然属于 `@snow/core`
- 延时任务的业务语义、覆盖逻辑、幂等逻辑都在 core 中实现
- `packages/web` 只提供一个可被 QStash 调用的 HTTP 回调入口
- 该回调入口只负责验签和调用 `@snow/core` 的 `executeIdleTasks()`，不承载 Snow 业务逻辑

也就是说：

- **调度逻辑在 Snow**
- **回调入口在 Web**

### 延时队列选型

- **延时队列：Upstash QStash**

### 覆盖逻辑

Batch 1 的延时任务语义保持和 M1 一致：

- 每次收到新消息，都创建一个“30 分钟后执行”的会话结束任务
- 如果旧任务还没有执行，就取消旧任务
- 保证同一用户始终只有一个待执行中的“会话结束任务”
- 只要 30 分钟内继续对话，就认为当前会话仍在持续
- 超过 30 分钟没有新消息，才认为会话结束，并执行 idle 收尾逻辑

换句话说：

- **30 分钟无消息 = 会话结束**
- **30 分钟内有新消息 = 会话继续，旧任务失效，推送新任务**

### 本批次顺带解决的 M1 技术债

1. 明确 Web 接入边界
   - 前端负责 UI 和配置来源
   - `@snow/core` 负责 Snow 核心逻辑
2. 明确 `getChatResponse()` 的外部注入方式
   - `platform = web`
   - `platformId` 稳定映射
   - `customDirective` 从 Web 显式传入
3. 明确当前消息历史在 Web 端的最小维护方式
4. `setTimeout` 只适合 CLI 常驻进程
5. 进程内 `Map` 覆盖逻辑不适合生产
6. 调度链路缺幂等约束
7. Idle 任务的失败恢复机制不完整

### 验证

```bash
pnpm --filter web dev
pnpm --filter web build

# ✅ Web 页面可访问
# ✅ Web 端能通过 /api/chat 调起 Snow
# ✅ 登录 / 注册可用
# ✅ 流式输出正常
# ✅ Vercel 可部署
# ✅ 新消息可覆盖旧延时任务
# ✅ 同一用户同一时刻只有一个待执行会话结束任务
# ✅ 30 分钟 idle 后只执行一次收尾任务
# ✅ Serverless 环境下依然正常
# ✅ 任务失败后可安全重试
# ✅ 回调重复触发时不会造成不可接受的重复副作用
```

## Batch 2：主动消息

### 目标

Snow 从“只会被动回复”变成“可以在合适时机主动来找你”。

### 任务

1. 定义主动消息触发条件
   - 长时间未聊
   - 用户低落后的回访
   - 某个事件的 follow-up
   - 关系维护 / 想念感表达
2. 定义主动消息数据流
   - 读取记忆 / 关系 / 情绪
   - 生成主动消息
   - 记录发送状态
3. 接入调度层
4. 支持基础防打扰策略
   - 夜间静默
   - 用户关闭主动消息
   - 不重复打扰

### 本批次顺带解决的 M1 技术债

1. 主动消息在 M1 只是设计存在，未实现
2. 调度层目前只覆盖 idle，不覆盖更丰富的事件触发
3. 缺少“消息已发 / 已取消 / 已失效”的状态管理

### 验证

```bash
# ✅ 满足条件时会创建主动消息任务
# ✅ 任务触发时能拿到正确上下文
# ✅ 不会重复向同一用户连续骚扰
# ✅ 用户关闭后不再发送
```

## Batch 3：图片消息支持

### 目标

Snow 支持理解用户发送的图片，不再只支持纯文本消息。

### 任务

1. 定义图片消息格式
2. 扩展消息处理层，支持图片内容提取
3. 在 Prompt 组装中注入图片描述
4. 支持图片相关的摘要与记忆写入
5. 处理图片消息的滑动窗口策略

### 本批次顺带解决的 M1 技术债

1. `ImagePart` 目前只是占位
2. 滑动窗口和 token 估算对多模态消息支持不完整
3. 图片上下文目前无法进入记忆链路

### 验证

```bash
# ✅ 用户发图片后，Snow 能理解并回应
# ✅ 图片关键信息可进入摘要
# ✅ 图片相关事实/语义内容可写入记忆
# ✅ 不会破坏当前文本消息链路
```
