## 一、设计哲学

### 1.1 核心命题

> Snow 不是一个聊天机器人，她是一个"人"。
> 让一个 AI 像人一样交流，核心不是语言能力——LLM 已经够好了。
> 核心是**记忆**。

一个人之所以像"人"，是因为她记得你、了解你、对你有独特的态度。
没有记忆，每次对话都是陌生人。有了记忆，才有关系、才有感情、才有"人味"。

### 1.2 第一性原理

**记忆是一切的基础设施。**

```
记忆系统
   ├── 驱动 → 关系评估（她和你是什么关系）
   ├── 驱动 → 情绪系统（她现在的心情）
   ├── 驱动 → 性格表达（她对你说话的方式）
   ├── 驱动 → 主动消息（她什么时候主动找你）
   └── 驱动 → 能力调用（她怎么帮你做事）
```

其他系统都是记忆的"下游消费者"。
记忆系统的质量直接决定了 Snow 的"人味"浓度。

## 二、核心循环

### 2.1 对话主循环

每一次用户发来消息，Snow 内部走过的完整流程：

```
用户消息到达
       │
       ▼
┌─────────────────────────┐
│  ① 记忆检索              │
│  从长期/短期记忆中检索     │
│  与当前话题相关的记忆      │
│                          │
│  输出：相关记忆列表       │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│  ② 关系评估              │
│  读取当前关系阶段和分数   │
│  决定 Snow 的态度基调     │
│                          │
│  输出：关系阶段 + 亲密度  │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│  ③ 情绪计算              │
│  基于用户消息 + 当前情绪  │
│  + 热上下文（摘要+最近几轮）│
│  计算新情绪              │
│                          │
│  输出：当前情绪状态       │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│  ④ Prompt 编排           │
│  组装最终 System Prompt： │
│  基础人设                 │
│  + 关系层                 │
│  + 用户自定义性格          │
│  + 情绪指令               │
│  + 记忆上下文             │
│  + 可用能力               │
│                          │
│  输出：完整 Prompt        │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│  ⑤ LLM 生成              │
│  调用大模型生成回复        │
│  流式输出给用户           │
│                          │
│  输出：Snow 的回复        │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│  ⑥ 记忆写入（异步）       │
│  onFinish 回调自动触发     │
│  push 到 Redis 待提取队列  │
│  每 5 轮增量提取           │
│  30 分钟延时任务兜底       │
│                          │
│  · 事实信息 → PostgreSQL  │
│  · 语义信息 → 向量数据库   │
│  · 摘要 → Redis（热）     │
│  ·      → PG（冷，延时）  │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│  ⑦ 关系更新（异步）       │
│  和记忆提取共用触发时机     │
│  LLM 分析 4 维信号增量     │
│  加权计算亲密度 → 判定阶段 │
│                          │
│  · 升级：自然过渡         │
│  · 降级：缓慢（1/3 速度） │
│  · owner：不降级          │
└─────────────────────────┘

┌─────────────────────────┐
│  ⑧ 情绪持久化（异步）     │
│  30 分钟 idle 触发         │
│  · 持久化情绪快照到 PG     │
│  · LLM 归纳情绪趋势摘要    │
│  · 供下次冷启动恢复        │
└─────────────────────────┘
```

### 2.2 为什么是这个顺序？

| 步骤 | 依赖 | 必须在前的原因 |
|------|------|---------------|
| ① 记忆检索 | 无 | 一切的起点，后续步骤都需要记忆上下文 |
| ② 关系评估 | ① | 关系状态影响 Prompt 中的人设层和态度 |
| ③ 情绪计算 | ① | 情绪需要结合热上下文（`context_summary + unextracted`）理解语境 |
| ④ Prompt 编排 | ①②③ | 需要所有上游结果才能组装完整 Prompt |
| ⑤ LLM 生成 | ④ | 必须等 Prompt 就绪 |
| ⑥ 记忆写入 | ⑤ | 需要知道 Snow 说了什么，才能完整记录本轮 |
| ⑦ 关系更新 | ⑤ | 需要完整对话才能评估关系信号 |
| ⑧ 情绪持久化 | ⑤ | 需要在会话结束后归纳本段情绪变化，不阻塞当前回复 |

**⑥⑦⑧ 是异步的**——不阻塞用户等待，回复先发出去，后台慢慢处理。

### 2.3 性能预期

| 步骤 | 预期耗时 | 是否阻塞用户 |
|------|----------|-------------|
| ① 记忆检索 | 100-300ms | 是（但可并行） |
| ② 关系评估 | < 10ms（读缓存） | 是 |
| ③ 情绪计算 | < 50ms | 是 |
| ④ Prompt 编排 | < 10ms | 是 |
| ⑤ LLM 生成 | 1-3s（流式） | 是（但流式输出，用户感知快） |
| ⑥ 记忆写入 | 500ms-2s | **否**（异步） |
| ⑦ 关系更新 | 200-500ms | **否**（异步） |
| ⑧ 情绪持久化 | 200-800ms | **否**（异步） |

用户感知延迟 ≈ ①+②+③+④ + ⑤首 token 时间 ≈ **0.5-1s 开始看到回复**

## 三、模块依赖关系

```
                    ┌──────────────┐
                    │   接入层      │
                    │ Web/QQ/微信   │
                    └──────┬───────┘
                           │
                           ▼
                    ┌──────────────┐
                    │  消息网关     │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
       ┌────────────┐ ┌────────┐ ┌──────────┐
       │ 记忆系统    │ │ 情绪   │ │ 关系     │
       │ (核心基建)  │ │ 系统   │ │ 系统     │
       └──┬───┬─────┘ └───┬────┘ └────┬─────┘
          │   │            │           │
          │   │      ┌─────┴───────────┘
          │   │      │
          ▼   ▼      ▼
       ┌──────────────────┐
       │  Prompt 编排引擎  │  ← 汇总所有上游信息
       └────────┬─────────┘
                │
                ▼
       ┌──────────────────┐
       │  LLM 调用层       │
       └────────┬─────────┘
                │
                ▼
       ┌──────────────────┐
       │  能力执行层       │  ← M1 暂不实现
       │  (MCP/Skill)     │
       └──────────────────┘
```

**依赖方向**：

| 模块 | 依赖 | 被依赖 |
|------|------|--------|
| 记忆系统 | 数据层（Supabase PG + pgvector + Upstash Redis） | 情绪、关系、Prompt 编排、主动消息 |
| 情绪系统 | 记忆系统 | Prompt 编排 |
| 关系系统 | 记忆系统 | Prompt 编排、主动消息频率 |
| Prompt 编排 | 记忆、情绪、关系、性格配置 | LLM 调用 |
| LLM 调用 | Prompt 编排 | 回复输出、记忆写入 |

**核心结论**：记忆系统是**唯一被所有模块依赖**的基础设施。它的设计质量决定了整个系统的天花板。

## 四、M1 架构决策

### 4.1 M1 = 记忆系统 + 基于记忆的人格表达

M1 的目标不是做一个"能聊天的 bot"，而是让核心循环跑通，让 Snow **活起来**。

### 4.2 M1 需要跑通的能力

| 能力 | M1 目标 | M2+ 增强 |
|------|---------|---------|
| 对话 | 流式对话，有性格 | 多轮优化 |
| 记忆读 | 每次对话前检索相关记忆 | RAG 语义检索优化 |
| 记忆写 | 对话后提取并存储关键信息 | 记忆衰减、冲突处理 |
| 关系 | 基于对话信号的关系评估和阶段判定 | 降级挽留、更细粒度信号 |
| 情绪 | 基于对话实时计算情绪状态 | 情绪惯性、EMA 平滑、会话结束后的趋势归纳 |
| 性格 | 三层人设 + Prompt 编排 | 用户自定义面板 |
| 前端 | 命令行脚本验证（core/scripts） | Web 聊天界面 |

### 4.3 M1 不做的事

| 暂不实现 | 原因 |
|----------|------|
| 主动消息 | 需要调度系统，M1 先做好被动对话 |
| MCP/Skill | 能力系统是锦上添花，先把"人"做好 |
| 多平台接入 | Web 先跑通，再铺其他平台 |
| 用户自定义面板 | M1 聊天中自然调整即可 |
| 向量数据库语义检索 | 已确认：M1 用 pgvector |

## 五、设计决策记录（ADR）

### ADR-001：记忆系统是核心基础设施

- **决策**：记忆系统的设计和实现优先级最高，其他模块围绕记忆构建
- **原因**：没有记忆就没有关系、没有情绪、没有个性化——Snow 就不是"人"
- **影响**：M1 的技术重心放在记忆的读写循环上

### ADR-002：用户自定义性格采用自然语言方案

- **决策**：用户通过自然语言描述定制性格，不使用固定参数滑块
- **原因**：参数滑块太机械，破坏"和一个人聊天"的体验
- **影响**：后端需要 LLM 解析用户意图 → 生成性格指令 → 注入 Prompt

### ADR-003：关系升降级采用多维信号评估

- **决策**：关系变化基于 5 个维度（互动频率、对话深度、情感浓度、信任信号、时间跨度）的加权评估
- **原因**：单靠"聊了几次"太机械，无法反映真实的人际关系动态
- **影响**：每次对话后需要 LLM 分析关系信号，异步更新

### ADR-004：表情策略仅使用 Emoji

- **决策**：M1 阶段仅使用 Emoji 表情，不做表情包图片
- **原因**：Emoji 足够表达情感，表情包需要 IP 设计和图片管理，复杂度高
- **影响**：在 Prompt 中引导 Snow 自然使用 Emoji

### ADR-005：记忆写入和关系更新异步执行

- **决策**：步骤 ⑥⑦ 不阻塞用户，回复先发出，后台处理
- **原因**：用户体验优先，记忆写入可能需要额外的 LLM 调用（500ms-2s）
- **影响**：使用 Upstash QStash 作为异步任务队列
