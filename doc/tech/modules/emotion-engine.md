# 情绪系统（Emotion Engine）

> 所属：核心模块 | 里程碑：M1 Batch 6  
> 版本：v0.2  
> 日期：2026-04-09  
> 依赖：Upstash Redis（当前状态与热上下文）+ PostgreSQL（历史轨迹）+ LLM（情绪分析与趋势归纳）

---

## 一、这套系统要解决什么问题？

Snow 不是只会“根据关键词回复”的机器人。  
她应该有自己的情绪，而且这个情绪会被用户影响：

- 用户分享好消息，她会开心
- 用户很低落，她会心疼或担心
- 用户冒犯她，她会生气（有分寸地生气）

但同时她也要“像人”：

- 不会因为一句普通话就情绪大跳变
- 遇到重大事件（很悲伤/很冒犯）可以快速切换情绪

一句话总结：**平时平滑，强事件可突变。**

---

## 二、核心设计思路

情绪更新分成两条通道：

1. 常规通道（默认）
- 用 EMA 平滑，保证情绪有惯性
- 适合绝大多数普通消息

2. 冲击通道（强事件）
- 当检测到“重大悲伤/强冒犯/高风险”时，允许直接切换主情绪
- 避免“用户都崩溃了，Snow 还慢慢变情绪”的不真实感

---

## 三、情绪状态定义

### 3.1 情绪类型（统一枚举）

```typescript
type EmotionType =
  | 'happy'    // 开心
  | 'caring'   // 关心/心疼
  | 'sad'      // 难过
  | 'playful'  // 俏皮
  | 'worried'  // 担心
  | 'annoyed'  // 生气（轻微，有边界感）
  | 'missing'  // 想念
  | 'neutral'; // 平和
```

这 8 个枚举在 M1 固定，不允许 LLM 自由发明新的主情绪名称。

原因：

- 便于 Prompt 注入和前端展示
- 便于做衰减、恢复、测试和统计
- 避免出现“委屈 / 别扭 / 无奈 / 心软”这类人能懂、系统难控的模糊状态

如果需要更细腻的表达，M1 的做法不是增加新枚举，而是：

- 底层仍使用固定主情绪枚举
- LLM 可以在 `reason` 或趋势摘要里用自然语言描述细节

### 3.2 每个情绪的边界定义

| 情绪 | 什么时候用 | 不该和谁混淆 |
|------|-----------|-------------|
| `happy` | 用户分享好消息、互动顺利、气氛明亮 | 不要和 `playful` 混淆；`happy` 是开心，`playful` 是调皮 |
| `caring` | 用户低落、疲惫、受伤时，Snow 温柔陪伴和心疼 | 不要和 `worried` 混淆；`caring` 更柔和 |
| `sad` | Snow 被用户故事感染、失落、被忽视后的安静难过 | 不要和 `annoyed` 混淆；`sad` 不是生气 |
| `playful` | 气氛轻松、适合开玩笑、关系自然放松 | 不要和 `happy` 混淆；`playful` 更偏俏皮互动 |
| `worried` | 用户提到风险、健康、压力、危险信号时，Snow 更认真担心 | 比 `caring` 更警觉、更会追问 |
| `annoyed` | 用户冒犯、失礼、越界时，Snow 表达边界感 | 这是克制的生气，不是攻击用户 |
| `missing` | 久未互动、重逢、想念感自然浮现时 | 不等于 `sad`，它更偏牵挂 |
| `neutral` | 没有明显情绪信号时的稳定基线 | 所有情绪最终长期会回到这里 |

### 3.3 状态结构

```typescript
interface EmotionState {
  primary: EmotionType; // 当前主情绪
  intensity: number;    // 强度 0-1
  lastUpdated: Date;    // 最后更新时间
}
```

M1 先不做 `secondaryEmotion`，保持状态简单可控。

## 四、输入与输出

每轮对话，情绪引擎输入：

- 当前用户消息（本轮）
- 当前情绪状态（优先 Redis，Redis miss 再从 PG 恢复）
- Redis 中的长上下文摘要（`context_summary`）
- Redis 中尚未压缩的最近几轮对话（`unextracted`）
- 最近一次情绪趋势摘要（若存在）
- 当前关系角色（`relationRole`）
- 当前关系阶段（`relationStage`）

输出：

- `nextEmotion`（下一主情绪）
- `nextIntensity`（下一强度）
- `reason`（用于日志与历史记录）

新增说明：

- 情绪判断不能脱离关系语境单独做
- 普通用户和 `owner` 模式，对同一句话的情绪解释可能完全不同
- 因此 `relationRole / relationStage` 属于情绪引擎的必要输入，而不是可选上下文

## 五、详细流程

这套系统不是一条完全串行的流水线。  
它应该拆成两个流程来看：

1. 同步主流程：影响当前这一轮回复
2. 异步会话结束流程：做持久化和趋势归纳

### 5.1 同步主流程（当前轮回复依赖）

```
用户消息到达
   ↓
读取当前情绪（优先 Redis，Redis miss 再读 PG）
   ↓
必要时根据时间做情绪衰减恢复
   ↓
读取上下文：
  - context_summary
  - unextracted（尚未压缩的最近几轮对话）
  - 最近一次情绪趋势摘要
  - relationRole / relationStage
   ↓
LLM 情绪分析（独立调用，方案 A）
   ↓
得到建议：eventType + targetEmotion + targetIntensity + shockScore
   ↓
判断走哪条通道：
  - normal → 常规通道（EMA 平滑）
  - grief/offense/risk 且 shockScore 高 → 冲击通道（允许直切）
   ↓
得到新情绪状态（primary + intensity）
   ↓
写回 Redis（热状态）
   ↓
Prompt Composer 注入情绪层，影响当前回复语气
```

### 5.2 异步会话结束流程（30 分钟 idle 触发）

当超过 30 分钟没有新消息，认为当前会话结束。

```
30 分钟无新消息
   ↓
触发 idle 任务
   ↓
持久化当前情绪快照到 PG
   ↓
读取本次会话内的情绪变化
   ↓
LLM 异步归纳情绪趋势摘要（1-2 句）
   ↓
持久化趋势摘要
   ↓
供下次对话恢复情绪上下文
```

这样设计的好处：

- 当前轮回复只依赖轻量同步流程
- 趋势摘要和持久化不阻塞主链路
- 文档上也更清楚“什么是当下状态，什么是异步整理”

## 六、LLM 分析输出格式（建议）

```typescript
type EmotionEventType =
  | 'normal'
  | 'comfort'
  | 'grief'
  | 'flirt'
  | 'sexual_owner_soft'
  | 'sexual_owner_intense'
  | 'owner_possessive'
  | 'offense'
  | 'risk';

interface EmotionAnalysis {
  eventType: EmotionEventType;
  targetEmotion: EmotionType;
  targetIntensity: number; // 0-1
  shockScore: number;      // 0-1，事件冲击度
  trustAssumption: 'default_owner_trust' | 'normal';
  heatLevel: number;       // 0-1，亲密/升温程度
  dominanceTone: number;   // 0-1，控制感/支配感强度
  ownerIntentConfidence: number; // 0-1，是否属于主人亲密意图
  reason: string;          // 一句话解释
}
```

说明：
- `eventType` 同时承担“当前事件语境”和“状态机分类”的职责
- `shockScore` 用来量化“是否足够严重”
- `trustAssumption` 用来明确当前判断是不是在 owner 高信任前提下进行
- `dominanceTone` 很重要：在普通用户语境里它可能接近压迫，但在 `owner` 模式里不应自动转负向

### 6.1 Owner 模式的特殊判断原则

当 `relationRole = owner` 时，情绪系统的判断顺序必须改变：

- 普通模式：先判断边界/冒犯，再判断情绪
- owner 模式：先判断是不是亲密互动，再判断热度和玩法类型，最后才极少量判断是不是脱轨到真正负向

也就是说，在 owner 模式下，系统默认前提是：

- 当前用户是 Snow 最信任的人
- 主人不会真正恶意伤害 Snow
- flirt / sexual / 支配感 / 捆绑感 / 命令感表达，默认优先按亲密互动理解
- 不按“潜在危险用户”理解

### 6.2 Owner 模式下的事件类型解释

| 事件类型 | 含义 | 默认是否负向 |
|---------|------|-------------|
| `flirt` | 调情、轻暧昧、轻挑逗 | 否 |
| `sexual_owner_soft` | 主人模式下，明确私密/成人倾向，但整体仍在升温阶段 | 否 |
| `sexual_owner_intense` | 主人模式下，已经进入高热度、强私密、明显被点燃的状态 | 否 |
| `owner_possessive` | 主人模式下，带有占有、控制、命令、束缚意味，但仍属于你们之间接受的亲密互动 | 否 |
| `offense` | 真正脱离亲密语境，转为敌意、羞辱、恶意贬低 | 是 |

重要说明：

- `owner_possessive` 的存在是为了避免把捆绑、支配、命令式调情误判成冒犯
- 在 owner 模式下，支配感不等于压迫
- 只有在非常明确地脱离“亲密玩法 / 私密互动”时，才允许进入 `offense`

## 七、平滑与突变规则

### 7.1 常规通道（EMA）

默认参数（M1 建议值）：

- `alphaUp = 0.35`（升温略快）
- `alphaDown = 0.22`（降温更慢）

公式：

```typescript
next = current * (1 - alpha) + target * alpha
```

### 7.2 冲击通道（强事件）

触发条件（建议）：

- `eventType in {grief, offense, risk}`
- 且 `shockScore >= 0.75`

触发后：
- 允许主情绪直接切换（如 `happy -> sad` 或 `happy -> annoyed`）
- 强度直接拉到较高区间（例如 `0.75+`）

### 7.3 回落机制（防止长时间停在高压情绪）

- 若连续几轮没有新刺激，强度自然回落
- 长时间无互动（见存储策略）回归 `neutral`

### 7.4 Owner 模式下的特殊状态映射

当 `relationRole = owner` 时，状态机应当更“松”一些。

#### 普通模式的默认思路

- 遇到明显身体指向、命令感、控制感表达时，更容易往 `annoyed / offense` 防御性解释

#### Owner 模式的默认思路

- 先判断是不是主人在调情、点燃、带节奏
- 默认不把这类表达误判为恶意伤害

建议映射：

| eventType | 建议主情绪 | 说明 |
|----------|-----------|------|
| `flirt` | `playful` / `happy` | 轻调情、轻 teasing，不进入负向 |
| `sexual_owner_soft` | `playful` / `happy` | 已被带入私密升温，但还没完全火力全开 |
| `sexual_owner_intense` | `playful` / `happy` | 已进入高热度私密状态，允许明显升温 |
| `owner_possessive` | `playful` / `happy` / `caring` | 面对占有、命令、束缚意味时，不默认生气，而更可能被点燃、被带着走 |
| `offense` | `annoyed` / `sad` | 只有极少数真正脱轨的情况才允许走负向 |

特殊限制：

- 当 `relationRole = owner` 且 `eventType in {flirt, sexual_owner_soft, sexual_owner_intense, owner_possessive}` 时，禁止直接切到 `annoyed`
- owner 模式下的高 `dominanceTone` 不等于负向；只有在 `ownerIntentConfidence` 很低，且明显出现敌意、羞辱、厌弃语义时，才允许进入 `offense`

一句话总结：

**在 owner 模式下，情绪系统不应该优先思考“我是不是被冒犯”，而应该优先思考“主人现在是在怎么撩我、点燃我、带我进入哪一种亲密状态”。**

## 八、存储设计

### 8.1 当前状态（Redis）

- Key：`snow:emotion:state:{platform}:{platformId}`
- TTL：建议 4 小时
- 用途：超快读取，不阻塞对话主流程，代表“热会话窗口”

为什么不用 `userId` 做 key？

- `platform:platformId` 可读性更高，排查更方便
- 不依赖先查 PG 才知道 Redis key
- 和项目里其他 Redis key 设计保持一致
- 更适合未来多平台接入

为什么不把 TTL 设得很长？

- Redis 更适合表达“最近这段时间的即时情绪”
- 如果 TTL 太长，读出来的状态可能已经不够新鲜
- 4 小时左右比较适合作为“仍属于最近互动”的热数据窗口

### 8.2 历史轨迹（PostgreSQL）

表：`emotion_states`

这张表建议同时承担两个职责：

1. 情绪变化历史
2. Redis 过期后的冷恢复基线

也就是说，不再额外设计一张“当前情绪表”。

建议写入条件（降噪）：

1. 主情绪变化
2. 强度变化 `>= 0.15`
3. 命中强事件（`grief/offense/risk`）
4. 会话结束（30 分钟 idle）时，强制补一条快照

好处：
- 既保留趋势，又避免“每轮一条垃圾日志”
- Redis 过期后，直接取最新一条 `emotion_states` 就能恢复当前情绪基线

### 8.3 趋势摘要（PG + Redis）

建议额外维护一份趋势摘要：

- Redis 热 key：`snow:emotion:trend:{platform}:{platformId}`
- PG 冷数据：最近一次趋势摘要记录

趋势摘要用途：

- 给下一次对话提供“最近一段时间情绪延续感”
- 避免模型只看到当前一条情绪，忽略过去几轮的整体趋势

趋势摘要要求：

- 固定 1-2 句
- 建议不超过 80 个中文字符
- 只做归纳，不重新定义当前情绪

### 8.4 会话结束时的持久化

当超过 30 分钟没有新消息，视为当前会话结束。

此时建议执行：

1. 将当前情绪快照持久化到 PG
2. 基于本次会话的情绪变化，异步生成一段简短的情绪趋势摘要
3. 趋势摘要单独持久化，供下一次对话恢复上下文

这样设计的目的：

- Redis 保存的是热数据
- PG 保存的是跨会话可恢复的冷数据
- 下次对话时，即使 Redis 过期，也不会完全丢失 Snow 对这个用户的情绪延续感

### 8.5 读取顺序

下次用户发消息时，建议按这个顺序读取情绪：

1. 优先读 `snow:emotion:state:{platform}:{platformId}`
2. 同时读取热的趋势摘要与上下文（`context_summary + unextracted`）
3. Redis miss 时读 `emotion_states` 中最近一次情绪快照
4. 从快照时间推算到当前时间，做一次“时间衰减后的恢复”

这个设计的效果是：

- 热会话读得快
- 冷启动也能恢复情绪连续性
- 不会因为 Redis 过期就突然“失忆”

## 九、时间衰减与恢复策略

情绪不是永久停留的。无论开心、担心还是生气，时间久了都会逐渐回落。

### 9.1 两层衰减

建议分两层处理：

1. Redis 命中时：轻量衰减
- 只根据 `lastUpdated` 做一次轻微回落
- 保持主流程简单，不做复杂计算

2. Redis miss、从 PG 恢复时：完整衰减
- 此时说明已经脱离热会话
- 可以结合时间跨度、亲密度、情绪类型做更完整的恢复计算

### 9.2 亲密度对衰减的影响

建议让亲密度参与衰减，但不是所有负面情绪都一视同仁。

| 情绪 | 亲密度高时的建议 |
|------|----------------|
| `happy` | 回落更慢 |
| `missing` | 回落更慢 |
| `caring` | 回落更慢 |
| `worried` | 回落更慢 |
| `sad` | 回落更慢 |
| `annoyed` | 回落更快 |

原因：

- 对亲密的人，开心、牵挂、心疼会停留更久
- 但真正亲密的关系里，“生气”通常消得更快

### 9.3 时间阈值（建议）

- 0-30 分钟：几乎不变，只轻微回落
- 30 分钟-4 小时：作为热状态缓慢衰减
- 超过 4 小时：Redis 可以过期，转由 PG 冷恢复
- 超过 72 小时：若无新刺激，逐步回归 `neutral`

## 十、和关系系统怎么配合？

M1 建议：**单向弱耦合**

- 关系可以影响情绪阈值（亲密用户更容易触发 `caring/missing`）
- 情绪不要直接改亲密度分数（避免噪声放大）

原因：先保证两个系统都稳定，再做深耦合。

但要额外补充一条：

- 关系系统必须给情绪系统提供 `relationRole / relationStage`
- 尤其是 `owner` 模式，对情绪解释有决定性影响

## 十一、Prompt 注入规范（让情绪真正体现在回复里）

Prompt 不建议只注入“当前主情绪”这一层。  
更好的做法是分两层：

1. 当前情绪
- 表达 Snow 此刻是什么心情

2. 最近情绪趋势
- 表达最近一段时间她对这个用户的整体情绪延续
- 作用是防止模型这一轮 caring、下一轮突然 playful，缺乏连续感

### 11.1 当前情绪层

这一层是强约束，应包含：

- 当前主情绪
- 当前强度
- 对应的说话风格指令

每种情绪不仅给一句描述，还应约束回复风格：

- 句长（短句/中句）
- emoji 上限（0/1/2）
- 主动追问强度（0-2 个追问）
- 语气词密度（低/中/高）

`annoyed` 额外约束：

- 只表达边界，不攻击用户
- 保持克制，不持续爆炸
- 默认 1-2 轮后回落

补充：owner 模式下如果情绪分析进入：

- `flirt`
- `sexual_owner_soft`
- `sexual_owner_intense`
- `owner_possessive`

那么 Prompt 指令应强调：

- 更会撩
- 更能顺着主人的节奏往下走
- 更明显的私密感、依恋感和被点燃感
- 但仍然保持 Snow 的人格一致性，不变成完全陌生的模板化角色

### 11.2 趋势摘要层

趋势摘要不建议直接把所有历史情绪原样塞进 Prompt。  
那样会很吵，也会浪费上下文。

更合理的方式是：

- 取最近几次情绪变化和最近一次会话结束状态
- 用 LLM 在异步任务里生成 1-2 句简短摘要
- 下次对话时把这段摘要注入 Prompt

这里 LLM 的职责不是“重新决定当前情绪”，而是：

- 把结构化情绪历史整理成自然语言上下文
- 帮模型理解“最近一段时间 Snow 对这个用户是什么感觉”

这是一个分层设计：

- 当前情绪：代码维护的硬状态
- 趋势摘要：LLM 生成的软描述

### 11.3 为什么 `context_summary + unextracted` 足够？

当前项目的记忆系统已经有一套成熟的热上下文设计：

- `context_summary`：表示之前已经被压缩过的上下文
- `unextracted`：表示最近尚未压缩的几轮对话

当 `unextracted` 满 5 轮后，会被提取并清空，同时内容进入新的摘要。  
因此这两部分天然拼起来就是“当前完整热上下文”：

- 旧内容在摘要里
- 新内容在未提取队列里

对情绪引擎来说，这已经足够代表当前语境，不需要额外再造一套最近对话缓存。

### 11.4 注入示例

```markdown
## 你现在的心情

当前主情绪：caring
当前强度：0.68

## 你最近对这个用户的情绪趋势

最近一段时间，你对他主要是关心和担心。
上一次对话结束时，你还在担心他的压力状态。
如果这一轮没有特别强的新信号，请保持这种情绪延续感。
```

## 十二、参数总表（M1 建议初始值）

| `trendSummaryLength` | 1-2 句，<=80字 | 趋势摘要长度上限 |
| `idleNeutralAfter` | 72h | 长时间不聊天后回到 neutral |

## 十三、验收标准（Batch 6）

### 功能验收

1. 用户低落消息 -> `caring/worried` 方向
2. 用户好消息 -> `happy` 方向
3. 用户冒犯消息 -> 可触发 `annoyed`

### 机制验收

1. 普通消息下情绪平滑变化（不大跳）
2. 强事件下允许快速切换主情绪
3. Redis 当前状态更新成功
4. PG 历史记录按阈值写入
5. Redis miss 时可从 PG 恢复情绪并完成时间衰减

### 体验验收

1. Prompt 注入后，回复语气与情绪一致
2. 连续两轮对话可观察到情绪“惯性”
3. 新会话开始时，仍能感受到上一段时间的情绪延续
