# AGENTS

本文件定义本项目的协作规则。所有后续开发任务默认遵循此文件。

## 1) 文档优先：必须先读 `doc/`

`doc/` 目录存储了项目核心文档，开始任何任务前必须优先阅读，并在实现后及时更新。

目录结构：

```text
doc/
├── README.md                # 文档导航
├── product/                 # 需求文档（PRD）
├── design/                  # 人设与交互设定
├── tech/                    # 技术选型、架构、模块设计
│   └── modules/             # 各模块实现设计
└── tasks/                   # 任务拆分、批次报告、里程碑记录
```

要求：
- 开始编码前，必须先阅读相关需求文档、技术文档、任务文档。
- 发现文档与代码不一致时，必须在任务中同步修正文档。
- 新增模块或关键决策时，必须在 `doc/` 下新增或更新对应文档。

## 2) 代码更新必须同步文档

每次代码变更都要检查是否影响以下内容，并及时更新：
- 需求范围（`doc/product/*`）
- 技术设计与实现口径（`doc/tech/*`）
- 任务状态与完成报告（`doc/tasks/*`）

禁止“只改代码不改文档”。

## 3) 需求流程：先讨论方案，再写文档，再写代码

每次新增需求或需求变更，必须遵守以下顺序：

1. 先与用户讨论并确认方案（目标、范围、取舍、验收标准）。
2. 更新或新建 `doc/` 中对应文档（需求/技术/任务）。
3. 基于确认后的文档实施代码开发。

禁止跳过文档直接编码。

## 4) 代码完成后必须做质量检查

每次完成功能或修复后，必须执行并修复 TypeScript 与 ESLint 问题：

- TypeScript 类型检查通过（无错误）
- ESLint 检查通过（无错误）

在可行情况下，提交前应确保相关脚本全绿。
